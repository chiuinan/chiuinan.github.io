字串轉ID演算法研究 － 邱奕南，2015/02/04

字串轉ID，本身也是搜尋的一種運用，只是因為比對運算相當耗時，因此字串
比對運算應儘量減少或避免。本文的目的，便是找出該運算最有效率的演算法。

傳統上，對於搜尋演算法，靜態資料主要是O(logn)的二分搜尋法（Binary Search），
以及O(n)的循序搜尋法（Sequential Search，儘適用於極少量資料）；動態資
料則是O(logn)的高度平衡樹，如AVL-Tree、Red-Black Tree等。但究極搜尋法，
則以O(1)的Hashing或Trie法為最快，這也是本文主要研究的兩大演算法。


一. Hashing

Hashing的效能，主要取決於下面兩個條件：

1.Hashing區要夠大
2.Hashing函數必須能將資料平均分佈於Hashing區

然而無論Hashing區有多大，Hashing函數如何設計，總會有某種資料集會造成
Hashing區資料分佈不均的情況發生（雖然機率不高）。此時碰撞（Collision）
的處理方式，便會嚴重影響到Hashing的效能。若Hashing區太小，或Hashing函
數分佈不均時，更是如此。


(一)碰撞處理演算法

首先我們先從碰撞的處理演算法開始探討起，這部份會決定Hashing所需的額外
空間大小。這些演算法大致有下列數種：

1.8 byte節點空間
  (1)隨機串列：最單純的處理，加入簡單，但搜尋必須整個掃瞄一遍，需時O(n)。
  (2)排序串列：搜尋到不可能時即可跳出，但加入必須要有排序好的上個節點，
     需時O(n)。
2.12 byte節點空間
  (1)隨機樹：依比對的結果直接加入樹中，最槽情況需時O(n)。
  (2)AVL樹：依比對的結果加入樹中，再進行遞迴式高度平衡，需時O(logn)。
3.16 byte節點空間
  (1)AVL樹：依比對的結果加入樹中，再進行非遞迴式高度平衡，需時O(logn)。
  (2)Splay樹：依比對的結果加入樹中，再將節點往最上層移動，並調整樹的
     平衡，平均需時O(logn)，最槽情況需時O(n)。

注意AVL樹有兩種寫法：遞迴與非遞迴。遞迴法所需節點空間較少，但高度平衡
運算時，必須做足logn次。非遞迴法可在高度未變化時即略去高度平衡運算，
但必須多一個父節點資料來進行上層回溯處理。基本上，後者的效能會較佳，
但也較耗記憶體空間。

從實測數據來看，當Hashing區遠大於資料數時，隨機串列的效能最好，但隨著
bucket資料數增多，它的效能便會變得非常差。至於bucket資料數增多時，建
立時間以Splay樹或隨機樹較佳，搜尋時間則以AVL樹較佳。綜合而言，碰撞處
理演算法應以遞迴AVL樹為最佳，原因為：

1.碰撞處理等同例外處理，當bucket資料較少時，各演算法的效能其實都相差
  不多，但萬一bucket突然有大量的資料進入時，碰撞處理的效能絕不可變得
  太差，因此首選便是O(logn)的AVL樹。隨機樹和Splay樹都有O(n)的最糟情況，
  因此基本上亦不應考慮。
2.搜尋處理上，建立只有一次，但搜尋可能多次。因此遞迴AVL樹與非遞迴AVL
  樹之間，應選擇搜尋較快的遞迴AVL樹，同時可節省較多的記憶體。雖然非遞
  迴AVL樹的建立時間較快，但一般情況下都是bucket資料數極少，這時兩者的
  建立時間是沒有差別的，因此並沒必要選擇非遞迴AVL樹。

除了AVL樹外，O(logn)的演算法還有紅黑樹（Red-Black Tree），它的特性為：

1.紅黑樹的高度平衡運算較少，建立時間比AVL樹快。
2.AVL樹高度最糟為1.44logn，紅黑樹則為2logn，因此AVL樹的搜尋速度會較快。

由於我們比較偏好搜尋速度較快的演算法，因此基本上亦不用考慮紅黑樹。


(二)Hashing函數與Key值函數

Hashing函數的主要目的，是儘量將輸入資料平均分散到Hashing區中。此處我
們挑選網評中備受讚賞的幾個Hashing函數：FNV-1、FNV-1a、Murmur3、CRC32，
來進行優劣比較，以決定最終採用的Hashing函數。

為了評比這些Hashing函數，個人定義了兩個評比指標：一是Bucket最多資料數，
好的Hashing函數不應將太多的資料塞到同一個Bucket中，此值愈小愈好；一是
資料搜尋平均比對次數，也就是要找到一個資料，平均需花多少次比對，當資

(三)動態Hashing區

Hashing搜尋法雖然具有更新快速、搜尋快速的優點，但卻有耗費大量記憶體的
缺點。例如設計目標是1億筆資料（128M），那麼Hashing區至少便要先配置512MB
以上，即使實際執行時，資料只有1千筆而已。如果有多種資料都需要用到Hashing
搜尋法，那麼很快就會把系統的記憶體全吃光光了。但如果為了節省記憶體，
將Hashing區縮得很小，那麼一但真的處理到設計目標的資料量，效能便會大幅
下降，並不符合原先預期的設計目標。此處我們要思考的是，如何在初始時只
配置微量的Hashing區，然後在執行過程中，有效率地讓Hashing區隨著資料量
的增加而擴大，但又不會嚴重地影響到整體效能。

首先想到的，是在碰撞時進行Rehashing處理。為了避免重複計算Hashing函數，
Rehashing使用的值應儘量取自最初的Hashing函數值。例如32位元Hashing函數
值，初始配置64K的Hashing區，使用掉16位元Hashing函數值，當有碰撞時，再
取8位元Hashing函數值並配置256的Rehashing區，若再有碰撞，便再取8位元
Hashing函數值並配置256的Rehashing區，最後再有碰撞，便做成AVL樹結構進
行搜尋。經過實驗，發現耗費的記憶體更多，效率反而變慢。主要是因為一個
資料碰撞便配置256的Rehashing區，最差情況會浪費255倍的空間。因此必須改
成每次只多取1位元Hashing函數值並配置2個Rehashing空間，如此最差情況只
會浪費一倍的空間。只是以位元為單位進行Rehashing，比對次數增多，效率可
能會變差不少。

另一種較簡單的做法，就是整個換掉重來。也就是重新配置更大的Hashing區，
再將已有的資料重新丟入。如果Hashing區是以2倍大小增長的話，處理的資料
量最多是2n，但因為不必重算Key值，因此其效率可以限縮在原處理時間的2倍
以內。以下便是這兩種方法處理1000萬筆資料的測試結果：

                 建立  搜尋  使用記憶體
------------------------------------------------
配足Hashing區    2.48  1.55  287MB
(1)碰撞Rehashing             216MB
  初始配置1K     6.65  5.51
  初始配置64K    5.00  4.88

的效能卻遠優於第一種方法，而且非常穩定，即便初始配置的Hashing區很小，
也不太會影響到效能。至於整體效能，也跟預期一樣，低於原處理時間的2倍以
內，算是非常適宜的方法。因此當處理的資料數已大於Hashing區大小時，便應
將Hashing區倍增，然後將所有資料重新Hashing處理。


(四)靜態資料Hashing

最後要考慮的是，當建完索引、處理完動態資料後，是否可轉變成更快速的搜
尋結構？首先想到的是，將動態存取的AVL樹，改變成已排序好的陣列資料形式，
然後再採用二分或循序搜尋法搜尋，在效率上似乎會更佳，而且也可省下不少
記憶體。於是我們針對這兩種結構進行實測，其結果為：

           AVL樹  排序陣列
----------------------------------
搜尋速度   1.55   1.98
使用空間   287MB  247MB

雖然使用空間變少了，但顯然地，採用排序陣列的搜尋速度並沒有比較快，反
而變慢了不少，因此並不值得做如此改變。

另一種想法，是將Hashing結構轉變成Trie結構進行搜尋，這牽涉到兩個部份：
一是轉變成Trie結構需時多久？因為Trie結構的建立速度很慢；一是Trie結構
的搜尋速度會比Hashing結構快多少？這些都必須等到Trie結構研究完畢後，才
能加以決定。

二. Trie

Trie結構的實作，現今大多採用雙陣列形式（Double Array Trie）。雖然Trie
結構的搜尋效率相當不錯，但它的建立速度非常差，為O(n^2)，只要資料量一
大，便會嚴重地影響到建索引的速度。以下我們先實測比對一下Hashing與Trie
這兩種結構的建立與搜尋速度：

                Hashing結構    Trie結構
資料            建立  搜尋    建立  搜尋
---------------------------------------------
500萬中文姓名   1.80  0.92  2770.8  0.76

果然和預期的一樣，Trie結構的搜尋速度較Hashing結構快，但它的建立速度，
真的是慢到會令人吐血！

由於我們已找出相當有效率的Hashing搜尋法，那麼能否就直接使用Hashing，
而捨棄Trie結構不用呢？答案其實是否定的，因為後切截搜尋必須依靠Trie結
構才能實現，其他像是拼字錯誤校正等，也都離不開Trie結構。此處，我們必
須徹底了解Trie結構的特性，以找出最快速建立Trie結構的方式。

在經過多方嘗試與實測後，發現Trie結構建立的速度，是隨著內部回收待用空
間的增加而變慢。而要使這些空間最少化，最佳的方式是一開始便將所有的資
料依著排序好的次序一一放入。以下為當資料都已排序好時，Hashing結構與
Trie結構的實測結果：

                Hashing結構    Trie結構
資料            建立  搜尋    建立  搜尋
--------------------------------------------
500萬中文姓名   1.31  0.59    3.01  0.19

與前述隨機資料的實測數據相比，Trie結構的建立效率相差將近1000倍，非常
可觀。不過到目前為止，我們只測試了中文姓名這類短字串，以下我們再測試
看看長字串資料，以及更大量的短字串資料，並觀察其記憶體的用量：

                    Hashing結構           Trie結構
資料            建立  搜尋  記憶體    建立  搜尋  記憶體
-------------------------------------------------------------
1000萬中文姓名  2.77  1.19   287MB    6.03  0.38   187MB
500萬長字串     4.68  3.28   579MB   33.35  3.10  3388MB

就短字串的建立時間而言，Trie結構其實還算不錯，但仍慢於Hashing結構，因
此如果不是需要大量重複搜尋的資料，其實使用Hashing結構即可。如果是需要
大量重複搜尋的話，那麼只要先將資料排序好，Trie結構會比Hashing結構更適
合。然而動態資料在收集完之前是無法排序的，此時便可善用Hashing搜尋法。
也就是先利用Hashing結構收集完動態資料後，再全部排序好，最後轉為Trie結
構。如果之前已有做成Trie結構，那麼便將新舊資料合併在一起排序好（可使
用Merge Sort），再重新製成Trie結構即可。

至於長字串的部份，從實測數據可以發現，Trie結構的建立速度與搜尋速度明
顯就慢了很多，而且還會耗用相當大量的記憶體。主要因為Trie結構是藉由整
併字串最前面相同的部份來節省記憶體，但每個資料byte需要使用8 byte來記
錄，也就是耗用記憶體會增長8倍，如果字串前面相同部份的比例太少（此為長
字串的特性），所需的記憶體便會暴增。因此對於長字串轉ID的處理，只能使
用Hashing結構。

前述的實測數據，是將已排序資料一一加入Trie結構中而成的。其實可依照
Double Array Trie的特性，從已排序的資料中直接抽出Trie結構，速度應該會
比逐一加入更快。但研究此一部份還需花不少時間，因此暫留待將來再繼續研
究改良。


三. 結論

依據前述實測所得結論寫成程式，在較慢的機器實測4638萬筆中文姓名資料，
結果如下（其中Trie的存檔時間含排序、合併、轉成Trie結構、壓縮等過程，
Hashing的存檔時間則只含壓縮）：

                  Hashing     Trie
------------------------------------------
建立3000萬筆
  建立時間        14.7        15.5
  耗用記憶體(MB)  1214～1288  1214～1288
  存檔時間        91.5        144.7
  資訊檔大小(MB)  266         303
  字串檔大小(MB)  173         102
  檔案總大小(MB)  439         405
搜尋
  讀檔時間        4.9         3.9
  搜尋時間        11.5        17.4
  耗用記憶體(MB)  1034        555
再增加1638萬筆
  建立時間        9.2         11.5
  耗用記憶體MB    1458～2033  1212～1222
  存檔時間        141.3       253.1
  資訊檔大小(MB)  419         456
  字串檔大小(MB)  268         320
  檔案總大小(MB)  687         776
搜尋
  讀檔時間        7.6         5.1
  搜尋時間        11.2       19.6
  耗用記憶體(MB)  1660        831

從實測數據看來，最耗時間的是存檔，其中有很大的比例是耗在資料壓縮上。
因此找出更快更有效率的資料壓縮方式（如LZ4），是下一個很重要的課題。不
過整體來看，效能算是不錯的了（隨機Trie結構，預估至少要24小時以上才能
處理完）。另外，合併（Merge）演算法目前使用的是最簡單的方式，其實亦可
改良成更快的Binary Merge法，但因這部份可縮減的時間很有限，並不是很急
迫的事，將來再視情況決定是否研究改良。

比較訝異的是，在高資料量下，Trie結構的搜尋速度並沒有比Hashing快，反而
變得更慢（可能是隨機存取的原因造成），因此除非特殊情況，最好還是選用
Hashing結構。不過Trie結構耗用的記憶體比Hashing少很多（對於短字串而言），
因此如果需要考量到記憶體耗用的情況，也可考慮採用Trie結構。

最後，我們將幾種能需要的字串轉ID處理的資料，做一個特性表，以決定出最
佳的處理方式。

資料        長度   數量   操作   處理方式
---------------------------------------------
英數字      短/中  巨     增     Trie
多字詞      短     巨     增     Trie
索引名稱    短/中  微     增刪   Hashing
文件名稱    中/長  巨     增刪   Hashing
來源名稱    長     中     增刪   Hashing
欄位名稱    短/中  微     增刪   Hashing
伺服器名稱  短/中  微     增刪   Hashing
使用者名稱  短/中  少     增刪   Hashing
群組名稱    短/中  少     增刪   Hashing
PDF CMap    短     少/中  增     Trie/Hashing
網站名稱    中/長  少     增     Hashing
濾除網址    長     多     增     Hashing

以下為各資料處理方式的考慮因素：

1.為支援後切截搜尋，英數字必須採用Trie結構。
2.抽詞時，需要搜尋所有匹配的連續詞，因此多字詞亦必須採用Trie結構。
3.資料長度在〔中〕以上，尤其是〔長〕者，應採用Hashing結構。
4.PDF CMap資料可考慮採用Trie結構，以節省記憶體，但採用Hashing亦無不可。
5.其他無特別需求者，均採用Hashing結構，尤其帶有刪除操作者。
