*** HTTPS

HTTPS協定，主要是將HTTP協定雙方應傳接的資料，透過SSL或TLS協定進行傳接，
同時將內定通訊埠由80改到443。SSL全名為Secure Sockets Layer，TLS全名為
Transport Layer Security，目前常用的有三種（參見IE的進階設定）：

    SSL 2.0
    SSL 3.0
    TLS 1.0

由於大部份的HTTPS伺服器都有支援SSL 3.0，但未必支援TLS 1.0，因此在實作
HTTPS Client時，可以SSL 3.0為主。事實上，TLS 1.0係由SSL 3.0微修而來，
一般亦稱為SSL 3.1，因此弄懂SSL 3.0後，要再擴增為TLS 1.0，其實是很簡單
的。因此以下說明的內容，便均以SSL 3.0為主。關於SSL 3.0與TLS 1.0的相關
資料，可參考：

DRAFT302: The SSL Protocol Version 3.0
RFC2246: The TLS Protocol Version 1.0

又，為了協助讀者看懂這些文件內容，後面的文章裡，我們都會先列出原文的
定義，然後再用比較容易看懂的方法，重新解說。

註：TLS 1.0新訂了一個RPF函數如下：

    PRF(secret,label,seed) = P_MD5(S1,label+seed) XOR P_SHA-1(S2,label+seed);

S1為serect前半字串，S2為serect後半，各半長度為(len+1)/2，亦即若為奇數
長度時，會有一個byte重疊。

    P_hash(secret,seed) = HMAC_hash(secret,A(1)+seed) + HMAC_hash(secret,A(2)+seed) + HMAC_hash(secret,A(3)+seed) + ...
    A(0) = seed
    A(i) = HMAC_hash(secret,A(i-1))
    HMAC_hash(secret,data)，secret為key值


*** 記錄層

SSL雙方傳接的資料，均拆分為Fragment，在SSL文件中定義為：

    struct {
        uint8 major, minor;
    } ProtocolVersion;

    enum {
        change_cipher_spec(20), alert(21), handshake(22),
        application_data(23), (255)
    } ContentType;

    struct {
        ContentType type;
        ProtocolVersion version;
        uint16 length;
        opaque fragment[SSLPlaintext.length];
    } SSLPlaintext;

亦即Fragment的結構為：

    Fragment資料形態：1 byte
    目前使用的SSL版本：2 byte，主版本在前，次版本在後，SSL 3.0應填入03 00
    Fragment資料大小：2 byte，假設值為n
    Fragment資料：n byte

Fragment資料形態有4種，一是啟動加密訊息(20=14h)，一是警示訊息(21=15h)，
一是Handshake訊息(22=16h)，一是應用程式資料(23=17h)，這些資料形態什麼
時候使用，如何使用，後面會再述及。

注意SSL裡的數字都是高位元組在前，因此16384 byte，即4000h byte時，Fragment
資料大小應填入的資料便是：

    40 00

Fragment資料可能是密文（當啟動加密後），也可能是明文（一開始時），前
面僅列了明文的定義，密文的部份，我們會在後面再述及。又，明文（Plain Text）
資料最長不能超過16384 byte，亦即n<=16384。


*** 邏輯層

SSL在開始傳接實際的應用程式資料前，必須先進行Handshake，以確認雙方採
用的SSL版本、加密演算法等資訊。其訊息傳遞的過程為：

1. Client端送出Client Hello訊息，告知Client端可支援的SSL版本、可支援的加密演算法
2. Server端送出Server Hello訊息，告知Server端可支援的SSL版本，以及決定採用的加密演算法
3. Server端送出憑證訊息或公鑰訊息，告知Server端使用的公鑰
4. Server端送出Server Hello Done訊息
5. Client端送出密鑰訊息（以公鑰加密），告知Server端密鑰的相關資訊
6. Client端送出啟動加密訊息
7. Client端送出Finished訊息（以密鑰加密）
8. Server端送出啟動加密訊息
9. Server端送出Finished訊息（以密鑰加密）
10. Client端與Server端開始互傳實際的應用程式資料

Handshake訊息的定義如下：

    enum {
        hello_request(0), client_hello(1), server_hello(2),
        certificate(11), server_key_exchange (12),
        certificate_request(13), server_hello_done(14),
        certificate_verify(15), client_key_exchange(16),
        finished(20), (255)
    } HandshakeType;

    struct {
        HandshakeType msg_type;
        uint24 length;
        select (HandshakeType) {
            case hello_request: HelloRequest;
            case client_hello: ClientHello;
            case server_hello: ServerHello;
            case certificate: Certificate;
            case server_key_exchange: ServerKeyExchange;
            case certificate_request: CertificateRequest;
            case server_hello_done: ServerHelloDone;
            case certificate_verify: CertificateVerify;
            case client_key_exchange: ClientKeyExchange;
            case finished: Finished;
        } body;
    } Handshake;

亦即Handshake訊息的結構為：

    Handshake訊息形態：1 byte
    Handshake訊息大小：3 byte（高位元組在前），假設值為n
    Handshake訊息資料：n byte

以下便依Handshake過程的順序，說明各Handshake訊息。


一. Client Hello（形態編號=1）

    struct {
        uint32 gmt_unix_time;
        opaque random_bytes[28];
    } Random;

    opaque SessionID<0..32>;

    uint8 CipherSuite[2];

    enum { null(0), (255) } CompressionMethod;

    struct {
        ProtocolVersion client_version;
        Random random;
        SessionID session_id;
        CipherSuite cipher_suites<2..2^16-1>;
        CompressionMethod compression_methods<1..2^8-1>;
    } ClientHello;

Client Hello的結構為：

    Client採用的SSL版本：2 byte，主版本在前，次版本在後
    Client Random：32 byte
        時間：4 byte，採Unix GMT格式，高位元組在前
        隨意亂數：28 byte
    Session ID：
        ID長度：1 byte，假設為a
        ID資料：a byte
    Client端可支援的加密法：
        加密法資料數：2 byte，高位元組在前，假設為b
        加密法編號：b byte，每2 byte一組，愈想用的排愈前面
    Client端可支援的壓縮法：
        壓縮法資料數：1 byte，假設為c
        壓縮法編號：c byte

若Client有支援TLS 1.0，SSL版本編號可填入03 01，這也就是為何TLS 1.0亦
可稱為SSL 3.1的原因。當Server亦支援TLS 1.0，在Server Hello時會回傳版
本3.1，此時雙方便可採用TLS 1.0協定。若Server只支援SSL 3.0，則Server Hello
會回傳版本3.0，如此雙方便必須採用SSL 3.0協定。當然，若Client只支援SSL 3.0，
這裡便應該填入03 00，如此Server自然必須使用SSL 3.0協定。

Client Random，在後面的Handshake訊息會用到，必須記錄下來。Unix GMT時
間格式，便是從1970/1/1 00:00:00開始到目前時間的秒數，這點可呼叫time(NULL)
函數，取得其回傳值便是。

Session ID是用來回復之前已通過Handshake的Session，直接採用裡面雙方已
同意的加密演算法與密鑰，避免再次Handshake，浪費時間。在Server Hello訊
息裡，會回傳一個Session ID，便是本次Handshake所處的Session，在雙方Handshake
成功，傳接完資料斷線後，Server會暫時將此Session記錄一段時間，若此時Client
端有需要而再次連線時，便可填入該Session ID，恢復啟用原先協定好的數據。
當然，在這種情況下，雙方傳接的Handshake訊息次序便和前述的標準流程有些
不同，這部份請自行參考相關文件。這裡，我們仍以第一次連線為準，因此應
填入ID長度為0。

Client端可支援的加密法，這部份較複雜些，我們先談壓縮法。在SSL 3.0裡的
壓縮法只有一種，就是不壓縮（編號=0），因此這裡應填入01 00。

注意所有SSL的資料都必須封裝成Fragment格式，因此第一次送給Server的Client Hello
訊息，實際送進socket裡的資料，應該為：

    ------ Fragment Header (5 byte) -----------
    16        // Handshake Fragment形態編號 //
    30 00     // SSL版本 //
    ?? ??     // Handshake訊息總長 = Client Hello訊息內容長度+4 //
    ------ Handshake訊息 Header (4 byte) ------
    01        // Client Hello訊息形態編號 //
    ?? ?? ??  // Client Hello訊息內容長度 //
    ------ Client Hello訊息 -------------------
    03 00     // SSL版本 //
    ....

當然，多個Handshake訊息亦可合併在同一Fragment裡，例如之後Server會連送
數個Handshake訊息，這些訊息可以集結在同一個Fragment裡，亦可拆成多個Fragment
分別傳送。總之，Fragment只是做為雙方傳接訊息的記錄格式而已，在邏輯上，
雙方皆只看Fragment裡面承載的資料而已。


二. Server Hello（形態編號=2）

    struct {
        ProtocolVersion server_version;
        Random random;
        SessionID session_id;
        CipherSuite cipher_suite;
        CompressionMethod compression_method;
    } ServerHello;

Server Hello的結構為：

    Server採用的SSL版本：2 byte，主版本在前，次版本在後
    Server Random：32 byte
        時間：4 byte，採Unix GMT格式，高位元組在前
        隨意亂數：28 byte
    Session ID：
        ID長度：1 byte，假設為a
        ID資料：a byte
    Server端選用的加密法編號：2 byte
    Server端選用的壓縮法編號：1 byte

Server Random，在後面的Handshake訊息會用到，必須記錄下來。

當決定了雙方所採行的加密法後，便可決定出幾個相關參數：

    CipherSpec.cipher_type：stream加密法或block加密法，RC4是stream加密法。
    CipherSpec.key_material：加密法的鍵值byte長度，RC4_128為16。
    CipherSpec.IV_size：加密法的Initial Vector byte長度，RC4為0（不使用）。
    CipherSpec.hash_size：digest的hash資料byte長度，MD5為16，SHA-1為20。
    CipherSpec.is_exportable：是否Exportable，會影響加密的鍵值產生法，RC4_128為false。

這些參數的意義，我們會在後面逐步提及。


三. Certificate憑證訊息（形態編號=3）

    opaque ASN.1Cert<1..2^24-1>;

    struct {
        ASN.1Cert certificate_list<1..2^24-1>;
    } Certificate;

Certificate的結構為：

    所有憑證資料總長：3 byte，高位元組在前
        第一張憑證資料長度：3 byte，高位元組在前，假設為a
        第一張憑證資料：a byte
        第二張憑證資料長度：3 byte，高位元組在前，假設為b
        第二張憑證資料：b byte
        ....

憑證資料主要是取得Server的公鑰，以便在傳送密鑰資訊（Client Key Exchange）
時，加密使用。各憑證資料皆為採用ANS.1結構DER編碼的X.509憑證，根憑證在
最後一張，Server憑證在第一張，因此解析驗證時，必須從最後一張解析起。
關於X.509憑證，我們會在後面再詳細說明，這裡只需要知道可取得Server的公
鑰即可。


四. Server Key Exchange公鑰訊息（形態編號=12）

Server Key Exchange係當前述憑證訊息裡的Server憑證，沒有可供密鑰資訊交
換時加密用的公鑰，臨時由Server提供的公鑰。以RSA而言，一般在正常的情況
下，是不太可能會遇到的，因此此處便略而不談。


五. Server Hello Done（形態編號=14）

    struct { } ServerHelloDone;

也就是訊息內容為空，故整個Handshake訊息固定為：

    0E 00 00 00


六. Client Key Exchange密鑰訊息（形態編號=16）

    struct {
        ProtocolVersion client_version;
        opaque random[46];
    } PreMasterSecret;

    struct {
        public-key-encrypted PreMasterSecret pre_master_secret;
    } EncryptedPreMasterSecret;

    struct {
        select (KeyExchangeAlgorithm) {
            case rsa: EncryptedPreMasterSecret;
            case diffie_hellman: ClientDiffieHellmanPublic;
        } exchange_keys;
    } ClientKeyExchange;

由於前面我們已限定了非對稱性加密法只支援RSA，故此處僅列出RSA相關的定
義。Client Key Exchange的結構為：

    pre_master_secret：48 byte
        雙方採用的SSL版本：2 byte，主版本在前，次版本在後
        隨意亂數：46 byte

注意pre_master_secret是要經RSA公鑰加密的（public-key-encrypted）。因
此實際的訊息內容長度並非48 byte，而是依公鑰的Modulus長度而定。RSA加解
密的部份，會在後面再述及。

有了pre_master_secret後，便必須配合前述的Client Random和Server Random，
產生出必要的Master Secret（固定48 byte）：

Master Secret = MD5(pre_master_secret + SHA('A' + pre_master_secret + ClientHello.random + ServerHello.random)) +
                MD5(pre_master_secret + SHA('BB' + pre_master_secret + ClientHello.random + ServerHello.random)) +
                MD5(pre_master_secret + SHA('CCC' + pre_master_secret + ClientHello.random + ServerHello.random));

也就是將字串"A"、pre_master_secret、Client Random、Server Random依次
送入SHA-1，得到20 byte digest，接著將pre_master_secret與前述20 byte digest
依次送入MD5，得到16 byte digest，做為Master Secret的前16 byte。然後再
做出第二與第三個16 byte，共同組成了48 byte Master Secret。Master Secret
資訊必須記錄下來，後面Handshake訊息會再用到（pre_master_secret不必）。

註：TLS 1.0改為Master Secret = PRF(pre_master_secret, "master secret", ClientHello.random+ServerHello.random)

Master Secret的作用，主要是用來產生加解密鍵值的相關資訊。首先做出block data
（每個block固定16 byte）：

    block data = block 1 data + block 2 data + block 3 data + ...

然後從block data裡，依次取出：

    client_write_MAC_secret[CipherSpec.hash_size]
    server_write_MAC_secret[CipherSpec.hash_size]
    client_write_key[CipherSpec.key_material]
    server_write_key[CipherSpec.key_material]
    client_write_IV[CipherSpec.IV_size]
    server_write_IV[CipherSpec.IV_size]

因此需要的block data總長為(hash_size+key_material+IV_size)*2。各block
資料的產生法為：

    block 1 data = MD5(master_secret + SHA('A' + master_secret + ServerHello.random + ClientHello.random))
    block 2 data = MD5(master_secret + SHA('BB' + master_secret + ServerHello.random + ClientHello.random))
    block 3 data = MD5(master_secret + SHA('CCC' + master_secret + ServerHello.random + ClientHello.random))
    block 4 data = MD5(master_secret + SHA('DDDD' + master_secret + ServerHello.random + ClientHello.random))

以此類推。以RC4_128、SHA1為例，共需(20+16+0)*2=72 byte，故至少需要產
生5個block data（80 byte）。

註：TLS 1.0改為block data = PRF(master_secret, "key expansion", ServerHello.random+ClientHello.random)

當CipherSpec.is_exportable為true時，還需額外進行下列運算：

    client_write_key = MD5(client_write_key + ClientHello.random + ServerHello.random);
    server_write_key = MD5(server_write_key + ServerHello.random + ClientHello.random);

由於RC4_128不是Exportable encryption algorithm，因此不必進行此一運算
（註：TLS 1.0亦有變更，請參見RFC2246）。

以Client端而言，當需要加解密時，在傳送時係以client_write_key、client_write_IV
做為鍵值進行加密，而以client_write_MAC_secret來產生核驗用的digest。在
接收時，則以server_write_key、server_write_IV做為鍵值進行解密，而以server_write_MAC_secret
來產生digest核驗。關於資料加解密的部份，我們會在下個訊息再詳細說明。


七. Change Cipher Spec啟動加密訊息（Fragment形態編號=20）

    struct {
        enum { change_cipher_spec(1), (255) } type;
    } ChangeCipherSpec;

Change Cipher Spec的結構為：

    change_cipher_spec：1 byte，固定值為1

由於Change Cipher Spec並非Handshake訊息，故必須獨立一個Fragment，因此
整個Fragment便會是：

    ------ Fragment Header (5 byte) -----------
    14        // Change Cipher Spec Fragment形態編號 //
    30 00     // SSL版本 //
    00 01     // Change Cipher Spece訊息總長 = 1 //
    ------ Change Cipher Spec訊息 -------------
    01        // Change Cipher Spece訊息 //

當Client端送出Change Cipher Spec訊息，表示之後傳送的訊息全部要開始加
密，而當Client端收到Server送來的Change Cipher Spec訊息，表示之後接收
的訊息全部要開始加密。以下為密文的定義：

    struct {
        ContentType type;
        ProtocolVersion version;
        uint16 length;
        opaque fragment[SSLCompressed.length];
    } SSLCompressed;

    stream-ciphered struct {
        opaque content[SSLCompressed.length];
        opaque MAC[CipherSpec.hash_size];
    } GenericStreamCipher;

    struct {
        ContentType type;
        ProtocolVersion version;
        uint16 length;
        select (CipherSpec.cipher_type) {
            case stream: GenericStreamCipher;
            case block: GenericBlockCipher;
        } fragment;
    } SSLCiphertext;

由於RC4是stream加密法，故此處我們只列出相關的定義。加密過程為：

1.假設要送出的明文資料為plain_text（不含Fragment Header）。
2.對plain_text進行壓縮成為compressed_text（因為SSL未支援任何壓縮法，
  故compressed_text=plain_text）。
3.計算MAC，附加到compressed_text後面，成為stream_cipher。
4.對stream_cipher加密，成為cipher_text。
5.將cipher_text填入Fragment裡送出。

解密過程則為：

1.接收一個Fragment，去除Fragment Header成為cipher_text。
2.對cipher_text解密，成為stream_cipher。
3.從stream_cipher取走舊MAC，成為compressed_text，並計算新MAC，核對新舊MAC是否相同。
4.將compressed_text解壓成為plain_text。
5.plain_text即為邏輯層應讀取的資料。

其中MAC的計算法為：

    MAC = hash(MAC_write_secret + pad2 +
               hash(MAC_write_secret + pad1 + seq_num + SSLCompressed.type + SSLCompressed.length + SSLCompressed.fragment));

hash為Server Hello訊息裡所決定應採用的hash法，如MD5、SHA-1等。pad1和
pad2與hash法有關：

    pad1 = MD5為連續48個36h字元，SHA-1為40個連續36h字元
    pad2 = MD5為連續48個5Ch字元，SHA-1為40個連續5Ch字元

註：TLS 1.0改為MAC = HMAC_hash(MAC_write_secret, seq_num + TLSCompressed.type + TLSCompressed.version + TLSCompressed.length + TLSCompressed.fragment));

MAC_write_secret以Client端而言，傳送時使用client_write_MAC_secret，接
收時使用server_write_MAC_secret（因為是Server端送出來的資料）。SSLCompressed.type
為Fragment形態編號，1 byte；SSLCompressed.length為compressed_text的長
度（不含MAC），2 byte。

其中必須要注意的是seq_num，為一8 byte整數（高位元組在前），用以記錄
Fragment編號（由0開始），可區分為send_seq_num與receive_seq_num。當送
出Change Cipher Spec訊息時，需將send_seq_num重設為0，當收到Change Cipher Spec
訊息時，需將receive_seq_num重設為0。每次傳送一個Fragment後，便將send_seq_num
加一，每次接收一個Fragment後，則將receive_seq_num加一。

另外，前面已提過，plain_text最大長度為16384 byte，因此當以stream加密
法加密後，cipher_text的最大長度便是16384+hash_size（不含5 byte Fragment Header）。


八. Client Finished（形態編號=20）

    enum { client(0x434C4E54), server(0x53525652) } Sender;

    struct {
        opaque md5_hash[16];
        opaque sha_hash[20];
    } Finished;

    md5_hash = MD5(master_secret + pad2 + MD5(handshake_messages + Sender + master_secret + pad1));
    sha_hash = SHA(master_secret + pad2 + SHA(handshake_messages + Sender + master_secret + pad1));

當Client端要送出Client Finished時，Fragment資料本身已需要加密，請參見
Change Cipher Spec的說明，這裡仍以明文的邏輯層來解說。pad1與pad2亦請
參見Change Cipher Spec。

Sender為送出本訊息者的編號（固定4 byte），Client Finished要用前者，
Server Finished要用後者：

    Client：43 4C 4E 54
    Server：53 52 56 52

handshake_messages為所有傳接到的Handshake訊息（明文），從Client Hello
開始，一直到Finished訊息之前（不含Finished訊息），按其傳接的次序組成。
Change Cipher Spec不是Handshake訊息，不可納入hash的計算。

註：TLS 1.0則為：

    struct {
        opaque verify_data[12];
    } Finished;

    PRF(master_secret, finished_label, MD5(handshake_messages) + SHA(handshake_messages)) [0..11];
    
    finished_label，Client端用"client finished"，Server端用"server finished"


九. Server Finished（形態編號=20）

當Client端收到Server Finished時，Fragment資料本身已需要解密，請參見
Change Cipher Spec的說明。Server Finished和Client Finished類似，請參
見Client Finished的說明。

十一. 應用程式資料（Fragment形態編號=23）

一般在Handshake後，雙方都已傳接了Change Cipher Spec訊息，故應用程式資
料雙方在傳接時，均有加密。請參見Change Cipher Spec的說明，其中的差別
僅在Fragment形態編號不同而已。

至於什麼時候斷線，或停止讀寫資料，係由外界按實際收到的資料自行決定，
與SSL協定無關。故SSL協定可運用於任何資料的傳接，包括HTTP、FTP、SMTP等
協定。


*** ANS.1結構與DER編碼

由於X.509憑證是採用ANS.1結構的DER編碼，故必須由此部份解說起。關於ANS.1
結構的邏輯定義，可參見：

X.680: Abstract Syntax Notation One (ASN.1): Specification of basic notation
X.681: Abstract Syntax Notation One (ASN.1): Information object specification
X.682: Abstract Syntax Notation One (ASN.1): Constraint specification
X.683: Abstract Syntax Notation One (ASN.1): Parameterization of ASN.1 specifications

而DER編碼部份，可參見：

X.690: ASN.1 encoding rules: Specification of Basic Encoding Rules (BER), Canonical Encoding Rules (CER) and Distinguished Encoding Rules (DER)

由於ANS.1結構的邏輯定義很繁瑣，因此這裡先暫時略過，直接說明DER編碼部
份，等到解說X.509憑證時，再按其出現的ANS.1結構定義進行說明。ANS.1結構
的DER編碼，其實和BER編碼類似，只是做了一些限制，使得同一個值的結構，
必定是唯一編碼。以下我們便以BER編碼為主進行說明，DER編碼的限制則穿插
其中。

ANS.1結構的BER編碼，主要由標記、長度、資料三個部份所組成。依次說明如下：


1.標記

標記的結構為：

    Class Type：2 bit，00 = Univerfal，01 = Application，10 = Context-specific，11 = Private
    Constructed：1 bit，0 = Primitive，1 = Constructed
    Tag編號：5 bit，0～30，31=使用擴充Tag編號
    擴充Tag編號：不定長度

Class Type一般均為Universal，除非有特別的定義，這部份我們會在解說X.509
憑證遇到時，再行說明。Constructed表示資料亦為ANS.1結構，必須再行解碼
成ANS.1結構；Primitive表示資料為實體資料，例如整數、字串等。

Tag編號當>=31時，必須採用擴充Tag編號法，從該byte開始，一直找到最高位
元不為1的byte，然後將各byte取低7位元，按其順序由高到低組合在一起。其
計算法為：

    int tag_number = *s & 0x1F;
    if (*s == 31)
    {
        tag_number = 0;
        for (;;)
        {
            ++s;
            tag_number = (tag_number << 7) | (*s & 0x7F);
            if ((*s & 0x80) == 0) break;
        }
    }
    ++s;


2.長度

長度的格式有短格式與長格式兩種，當第一位元組最高位元為0時為短格式，其
餘7位元便是長度值，若最高位元為1則為長格式，其餘7位元表示其後接幾個位
元組，共同組成長度值（高位元組在前）。其計算法為：

    if (*s & 0x80)
    {
        int no = *s & 0x7F;
        length = 0;
        for (int i=0; i < no; ++i)
        {
            ++s;
            length = (length << 8) | (unsigned char)*s;
        }
    }
    else
    {
        length = *s;
    }
    ++s;

當長度為0，且為Constructed時，表示不定長度，實際子結構的終止以EOC為準
（後述）。DER編碼不允許不定長度，故長度為0，便表示裡面真的沒資料。


3.資料

總byte數由長度值而定。是否應繼續展開子結構，亦由標記中的Constructed決定。


Class Type為Universal時，已有許多固定意義的Tag編號：

    EOC               0
    BOOLEAN           1
    INTEGER           2
    BIT STRING        3
    OCTET STRING      4
    NULL              5
    OBJECT IDENTIFIER 6
    OBJECT DESCRIPTOR 7
    EXTERNAL          8
    REAL              9
    ENUMERATED        10
    EMBEDDED PDV      11
    UTF8 STRING       12
    RELATIVE OID      13
    SEQUENCE          16
    SEQUENCE OF       16
    SET               17
    SET OF            17
    NUMERIC STRING    18
    PRINTABLE STRING  19
    T61 STRING        20
    VIDEOTEX STRING   21
    IA5 STRING        22
    UTC TIME          23
    GENERALIZED TIME  24
    GRAPHIC STRING    25
    VISIBLE STRING    26
    GENERAL STRING    27
    UNIVERSAL STRING  28
    CHARACTER STRING  29
    BMP STRING        30

簡單說明幾個（特別是後面X.509憑證會用到的）：

1.EOC

做為不定長度子結構的終止結構（DER編碼用不到），Primitive，內容為空，
故其編碼為：

    00 00

2.BOOLEAN

布林值，Primitive，0為false，非0為true，DER編碼限制true必須為1。故其
編碼為：

    false：01 01 00
    true ：01 01 01

3.INTEGER

整數，Primitive，最高位元組在前，第一個位元組最高位元表示數字的正負號。

4.BIT STRING

位元字串，Primitive，第一位元組表示最後位元組的最低位元有多少個不使用，
0表全部用到。DER編碼要求不用的位元必須清為0。

5.NULL

空值，Primitive，內容為空，故其編碼為：

    05 00

其他的Tag編號與內容結構，留待X.509憑證遇到時，再行解說。


*** X.509憑證

關於X.509憑證，可參見：

RFC5280: Internet X.509 Public Key Infrastructure Certificate and Certificate Revocation List (CRL) Profile

由於X.509憑證是採用ANS.1結構的DER編碼，故必須先行解碼成ANS.1結構，方
能解析。以下的說明，均以ANS.1的邏輯結構為主進行說明，部份再輔以實際的
編碼值，以釐清觀念。

    Certificate  ::=  SEQUENCE  {
        tbsCertificate       TBSCertificate,
        signatureAlgorithm   AlgorithmIdentifier,
        signatureValue       BIT STRING  }

這是X.509憑證一開始的定義，亦即它是一個SEQUENCE結構，裡面包含了三個子
結構。因此DER解碼後的ANS.1結構，必須只有一個，標記應為Constructed Universal SEQUENCE
（Tag編號為16），其編碼為30h（參見前述ANS.1結構與DER編碼），因此憑證
資料的第一個byte必定是30h。由於是Constructed，故SEQUENCE裡的資料，必
須再繼續解碼成ANS.1結構（必定為3個ANS.1結構），才能繼續解析。

由於要分析完整個X.509憑證，其實是很繁瑣的一件事，裡面有些資料對於SSL
協定而言，並沒有太大意義，因此以下我們只針對重要的部份進行解說，有興
趣的人可參考前列的資料，詳細了解其他被略過的資訊。在本文裡，我們主要
針對下列幾個部份的相關內容進行說明：

1.取得憑證內的公鑰
2.是否為CA憑證，亦即憑證內的公鑰是否可用來簽證其他憑證
3.憑證內的公鑰是否可用來做SSL密鑰資訊交換加密
4.憑證的簽證者（Issuer）與公私鑰擁有者（Subject），以判斷是否自我簽證
5.數位簽章的核驗

在前述的定義裡，tbsCertificate便是憑證內容，前4項所需的資訊都在這裡面，
signatureValue為數位簽章內容，signatureAlgorithm則為數位簽章採用的演
算法，這兩個是第5項所需的資訊。以下我們便從tbsCertificate憑證內容開始
逐步解析起。

    TBSCertificate  ::=  SEQUENCE  {
        version         [0]  EXPLICIT Version DEFAULT v1,
        serialNumber         CertificateSerialNumber,
        signature            AlgorithmIdentifier,
        issuer               Name,
        validity             Validity,
        subject              Name,
        subjectPublicKeyInfo SubjectPublicKeyInfo,
        issuerUniqueID  [1]  IMPLICIT UniqueIdentifier OPTIONAL
        subjectUniqueID [2]  IMPLICIT UniqueIdentifier OPTIONAL
        extensions      [3]  EXPLICIT Extensions OPTIONAL
    }

TBSCertificate同樣是一個Constructed Universal SEQUENCE結構，因此標記
編碼同樣是30h。不過裡面子結構的定義就比較多了，我們先說明各子結構的意
義：

    version：憑證的版本，目前有版本v1，v2，v3
    serialNumber：憑證的序號
    signature：電子簽章採用的演算法，必須跟前述的signatureAlgorithm內容相同
    issuer：憑證簽證者
    validity：憑證的有效期間
    subject：公私鑰擁有者
    subjectPublicKeyInfo：公鑰資訊
    issuerUniqueID：憑證簽證者的ID，必須v2或v3才有
    subjectUniqueID：公私鑰擁有者的ID，必須v2或v3才有
    extensions：擴充定義，必須v3才有

1.version（憑證版本）

    version [0]  EXPLICIT Version DEFAULT v1

這樣的ANS.1文法稱為Context-specific，因此Class Type需使用Context-specific，
Tag編號為0，且必須是Constructed，故編碼值為A0h。當省略時，其內定值為
版本v1。

    Version  ::=  INTEGER  {  v1(0), v2(1), v3(2)  }

在前述Constructed Context-specific Tag 0結構裡，只會有一個子結構，就
是Primitive Universal INTEGER，其資料內容（整數）便是版本的編號，0為v1，
1為v2，2為v3。

憑證版本除了影響TBSCertificate裡結構成員的有無外，同時也影響是否可用
為CA憑證。v1或v2的憑證，都是CA憑證，v3憑證除非特別在extensions裡標明，
否則一律不做為CA憑證。檢測是否為CA憑證的目的，我們會在數位簽章核驗時
再做說明。


2.serialNumber（憑證序號）

    CertificateSerialNumber  ::=  INTEGER

憑證序號是一個Primitive Universal INTEGER結構，其內容對於SSL協定而言

7.subjectPublicKeyInfo（公鑰資訊）

    SubjectPublicKeyInfo  ::=  SEQUENCE  {
        algorithm            AlgorithmIdentifier,
        subjectPublicKey     BIT STRING  }

SubjectPublicKeyInfo是一個Constructed Universal SEQUENCE結構，裡面包
含兩個結構：

    algorithm：公鑰適用的演算法
    subjectPublicKey：公鑰資料

AlgorithmIdentifier在前面signature裡已提過，但要注意這裡的OID不應包含
hash function，因此必須為：

    rsaEncryption OBJECT IDENTIFIER ::= { pkcs-1 1 }

亦即：

    rsaEncryption = 2A 86 48 86 F7 0D 01 01 01

    RSAPublicKey ::= SEQUENCE {
        modulus INTEGER, -- n
        publicExponent INTEGER -- e }

故必須解碼後，方能取得RSA加解密所需的Modulus與Public Exponent。又因
ANS.1裡的整數是有正負，而RSA的Modulus是無號數，因此若最高位元為1時，
ANS.1的整數資料會在最前面多加一個byte 00，以避免被解譯成負數。故取得
Modulus時，應去除所有前導0，方為Modulus真正的位數。關於RSA加解密的部
份，我們會在後面再說明。

1.Key Usage

先從extnValue（Primitive Universal OCTET STRING）取得資料內容後，視為
ANS.1結構的DER編碼，解碼回ANS.1結構：

    KeyUsage ::= BIT STRING {
        digitalSignature        (0),
        nonRepudiation          (1),

        cRLSign                 (6),
        encipherOnly            (7),
        decipherOnly            (8) }

解析得的結構應為一個Primitive Universal BIT STRING結構，取得位元資料

    keyEncipherment：如果為1的話，便可以用來做為SSL密鑰資訊交換加密使用，否則便不行。

        pathLenConstraint       INTEGER (0..MAX) OPTIONAL }

為一Constructed Universal SEQUENCE結構，裡面含有兩個結構，我們關心的

憑證。注意cA若是省略，內定為false。


*** X.509憑證電子簽章核驗

在說明電子簽章如何核驗之前，我們要先決定是否可核驗。在SSL協定裡，我們
可得到一堆X.509憑證，解析順序為從最後一張開始。通常前一張是下一張的簽
證者，但並非絕對，因此必須先進行下列判斷：

1.本憑證內的Issuer（簽證者）和Subject（擁有者）是否相同？若相同表示是
  自我簽證，以本憑證的公鑰進行電子簽章核驗。
2.有前一張憑證時，若該憑證為CA憑證，且前一張憑證的Subject就是本憑證的
  Issuer，表示本憑證為前一張憑證所簽證，以前一張憑證的公鑰進行電子簽
  章核驗。
3.找不到任何簽證者，不進行電子簽章核驗。

又，當電子簽章核驗全數通過後，必須檢查最後一張（亦即整個憑證列的第一
張，Server憑證）的公鑰是否可用來做為SSL密鑰資訊交換加密使用，如果不行
便是有問題。

得到了簽證者的公鑰後，便可以開始進行電子簽章核驗。首先取得signatureValue
的Bit String內容，其無用位元必須為0。然後以RSA公鑰解密後，視同ANS.1結
構DER編碼，解碼回ANS.1結構（參見RFC2313）：

    DigestInfo ::= SEQUENCE {
        digestAlgorithm DigestAlgorithmIdentifier,
        digest Digest }

    DigestAlgorithmIdentifier ::= AlgorithmIdentifier

    Digest ::= OCTET STRING

AlgorithmIdentifier前面已提過，只不過這裡的OID又有點不同了，必須為：

    digestAlgorithm OBJECT IDENTIFIER ::= { iso(1) member-body(2) US(840) rsadsi(113549) 2 }
        md2 OBJECT IDENTIFIER ::= { digestAlgorithm 2 }
        md4 OBJECT IDENTIFIER ::= { digestAlgorithm 4 }
        md5 OBJECT IDENTIFIER ::= { digestAlgorithm 5 }
    sha1 OBJECT IDENTIFIER ::= { iso(1) identified-organization(3) oiw(14) secsig(3) algorithms(2) 26 }

以下則是OID的編碼（MD4用不到）：

    digestAlgorithm = 2A 86 48 86 F7 0D 02
    md2             = 2A 86 48 86 F7 0D 02 02
    md4             = 2A 86 48 86 F7 0D 02 04
    md5             = 2A 86 48 86 F7 0D 02 05
    sha1            = 2B 0E 03 02 1A

OID代表的hash function必須和signatureAlgorithm中所指定的hash function
相同。實際應驗證的為digest（Primitive Universal OCTET STRING結構），
取得其資料內容，便是應檢驗的digest值。接著採用相同的hash function，將
tbsCertificate的DER編碼資料送入，若兩個digest值相同，核驗便算通過。


*** RSA

RSA是1977年由Ron Rivest、Adi Shamir、Leonard Adleman三人所提出的，於
是便取其三人名字的字首，做為這個加密法的名稱。RSA加密法，主要是根據模
數運算（Modular Arithmetic）的特性所發展出來的，以下便是相關的做法：

1.公私鑰的建立
  (1)找任意兩個質數p和q，p不等於q，計算n=p*q
  (2)選擇一個大於1小於n的自然數e，e必須與p-1和q-1互質（一般實務上多挑65537）
  (3)找出d，使得d*e mod (p-1)*(q-1) = 1
  (4)n,d為私鑰，n,e為公鑰（n=modulus，e=public exponent，d=private exponent）

2.加密
  (1)假設訊息為a，先轉為小於n的整數x
  (2)計算y = x^e mod n（公鑰加密）或y = x^d mod n（私鑰加密）
  (3)將整數y轉為訊息b

3.解密
  (1)假設訊息為b，先轉為整數y
  (2)計算x = y^d mod n（私鑰解密）或x = y^e mod n（公鑰解密）
  (3)將整數x轉回訊息a

亦即RSA係利用x^(d*e) mod n = x的特性來進行加解密的。它的保密性質在於，
當n夠大時，任何人無法在有限的時間內，將它分解成質因數。亦即即使知道n，
亦無法反推回p與q，自然就算不出d值來破解。如果有人能夠提出有效的質因數
分解法，那麼RSA便能輕易地破解，自然也就沒有存在的意義了。至於n值需要
多大？目前一般的應用，多半要求n值在1024位元以上。


*** PKCS#1標準

關於RSA訊息轉為整數的做法，現行皆採用PKCS#1標準，相關資料可參考：

RFC2313: PKCS #1: RSA Encryption Version 1.5

其格式為：

    00 || BT || PS || 00 || D

第一個byte 00，是為確保所得數字，必定小於模數（Modulus）n。BT為Block Type，
私鑰加密可為00或01，公鑰加密必須為02。PS為附加字串（Padding String），
至少要8 byte，這部份後面再述及。D則為訊息資料。

由於整個格式的長度是size(D)+11 byte以上，因此模數n至少要有12 byte（96 bit）
以上。由於目前n值多半要求1024 bit，也就是128 byte以上，因此這部份不成
問題。

1.Block Type 0

其Padding String所加入的字元固定為00，由於和分隔用的00相同，因此訊息
資料D的第一個byte必須不是0，或是雙方已知應傳接資料的大小。也因為有這
個限制，Block Type 0變成愈來愈少人使用。

2.Block Type 1

其Padding String所加入的字元固定為FFh，由於和分隔用的00不同，因此很容
易可以區分出訊息資料D。

3.Block Type 2

Padding String必須是非0的亂數，由於和分隔用的00不同，因此也是很容易可
以區分出訊息資料D。


*** 大整數運算

由於RSA是基於大整數的運算，無法直接使用系統的整數乘除運算，必須使用陣
列來模擬，因此大整數的運算速度，便成為RSA效能的瓶頸。由於大整數的運算
速度一般都不快，故RSA很少用來直接加密資料，多半用來交換其他運算速度較
快的加密演算法鍵值。在大整數運算中，最需要的便是乘法和除法（餘數運算同）：

1.乘法

兩個大整數相乘，最簡單的做法便是採用傳統的移位累加法，亦即：

             7 3
     *       2 9
    -------------
             2 7 (9*3)
           6 3   (9*7)
           0 6   (2*3)
     +   1 4     (2*7)
    -------------
         2 1 1 7

其效率是O(n^2)。一種簡單的改良法，便是使用分割解決法。假設兩數x和y要
相乘：

    x = a*2^n + b
    y = c*2^n + d

則：

    u = (a+b)*(c+d)
    v = a*c
    w = b*d
    z = v*2^2n + (u-v-w)*2^n + w

使用3次乘法來取代4次乘法，其效率為O(n^log3)=O(n^1.59)。其他還有Toom-Cook
的多重分割解決法，效率為O(n^(1+3.5/sqrt(logn)))，以及DFT法，效率為
O(nlognloglogn)，不過皆有其限制。

2.除法

兩個大整數相除，最簡單的做法便是採用傳統的移位累減法，亦即：

                2 5 ------> 商
         +-------------
    2 9  |    7 3 0
           -) 5 8
          -------------
              1 5 0
           -) 1 4 5
          -------------
                  5 ------> 餘數

其效率為O(n^2)。其他有Reciprocal法（牛頓逼近法），效率同乘法演算法，
但亦有其限制。


目前已知OpenSSL乘法採用分割解決法，除法採用移位累減法。關於大整數運算
的部份，將另以專文探討。


*** RSA加解密演算法

RSA主要的加解密運算在於次方運算。對於公鑰加解密而言：

     x^e mod n

若直接以x乘上e次，時間會需要非常久，一般皆採用指數位元累乘法來處理，
其演算法為（e>1，xq，則：

    dP = (1/e) mod (p-1) = d mod (p-1)，即dP*e mod (p-1) = 1
    dQ = (1/e) mod (q-1) = d mod (q-1)，即dQ*e mod (q-1) = 1
    qInv = (1/q) mod p，即qInv*q mod p = 1

dP與dQ稱為CRT exponents，qInv稱為CRT coefficient，這些多在產生公私鑰
時，即已算好並置入私鑰資訊裡。於是要計算y^d mod n，則：

    m1 = y^dP mod p
    m2 = y^dQ mod q
    h = qInv*(m1-m2) mod p  // h必須是正值，m1-m2可能是負值 //
    x = m2 + h*q

關於模數倒數（Modular Inverse）的計算，我們後面會再述及。


*** 大質數產生法

RSA鍵值產生的第一個要件，便是找出兩個大質數，但如何確定選用的大整數是
質數，其實是一件很困難的事。在小質數下，一般多採用試除法：

    // 假設質數2與其倍數均已排除在外 //
    n = sqrt(p);
    for (int i=3; i < =n; i+=2)
        if (p % i == 0) then p不是質數

想當然耳，試除法用在大整數上，根本不可行，否則模數n值便可用試除法分解
出p和q了。目前唯一能在Polynomial時間內，完全確認給定數字是否為質數的
方法，便是2002年被提出的AKS質數檢驗法，這部份請自行參閱相關資料。

在實務上，大質數的產生，多以檢測法將其可能是非質數的機率降到一定程度
以下。其過程為：

1.使用亂數產生一個待測的可能大質數。
2.由於RSA鍵值要求e與(質數-1)互質，故可先行檢驗，若非互質，則回步驟1。
3.利用質數檢測法進行多次測試，若均通過，即可將之視為質數，否則回步驟1。

首先便是如何有效率地產生一個待測的可能大質數，如果隨便產生，當然很容
易就是非質數。按質數理論，質數分佈的密度，亦即小於n的質數個數約為n/loge(n)，
而且愈接近1的地方分佈愈密集。因此當以亂數產生一個大整數後，可再利用埃
拉托斯特尼（Eratosthenes）篩選法，將該亂數移到可能的質數帶中。其演算
法為：

1.用亂數產生一個奇數的大整數p（排除質因數為2的情況）。
2.準備k個小質數（2除外），計算餘數r[i] = p mod prime[i], for i = 1～k
3.令delta = 0
4.若有任何餘數r[i]+delta mod prime[i] = 0，則delta += 2（需考慮Overflow），繼續步驟4
5.以p+delta做為待測的質數

*** 兩數互質檢測法

兩數互質，也就是GCD(a,b)=1，一般可用輾轉相除法，其效率為O(n^2)，n=位數。
演算法如下：

    r = a % b，if r = 1 then 互質，if r = 0 then 不互質
    gcd(a,b) = gcd(b,r)


*** 模數倒數（Modular Inverse）演算法

若a*b mod n = 1，則a與b在模數n中互為模數倒數，一般亦可寫成：

    a = 1/b mod n  或
    a = b^-1 mod n

模數倒數在RSA裡，計算d、qInv值時都需要，一般多採用Extended Euclidean algorithm：

function extended_gcd(a,b) // 求ax + by = GCD(a,b), 回傳(x,y,gcd) //
    x := 0    lastx := 1
    y := 1    lasty := 0
    while b ≠ 0
        temp := b
        quotient := a div b
        b := a mod b
        a := temp
        temp := x
        x := lastx-quotient*x
        lastx := temp
        temp := y
        y := lasty-quotient*y
        lasty := temp
    return {lastx, lasty, a}

以gcd(b,n)代入，回傳的lastx便是模數倒數，其中gcd必須為1。注意回傳的
lastx可能是負值，故必須判斷並轉成正值。


*** RC4

RC4全名為Rivest Cipher 4，為提出RSA加密法作者之一的Ron Rivest於1987所
提出。相關資料與演算法可自行上網搜尋查閱，這裡我們直接列出其程式供參考：

class RC4    // RC4加解密物件 //
{
private:
    int m_X, m_Y;
    unsigned char m_State[256];
public:
    void SetKey(const void* key, int key_len);
    void Encrypt(const void* data, int len, void* output);
};

void RC4::SetKey(const void* key, int key_len)
//說明：設定鍵值
//輸入：key, key_len = 鍵值資料與大小
{
    m_X = 0;
    m_Y = 0;
    int i;
    for (i=0; i < 256; i++) m_State[i] = i;
    unsigned char index1 = 0;
    unsigned char index2 = 0;
    for (i=0; i < 256; i++) 
    {
        index2 = (((char*)key)[index1] + m_State[i] + index2) & 0xff;
        char t = m_State[i];
        m_State[i] = m_State[index2];
        m_State[index2] = t;
        index1 = (char)((index1 + 1) % key_len);
    }
}

void RC4::Encrypt(const void* indata, int in_len, void* outdata)
//說明：對輸入資料進行RC4加解密
//輸入：indata,in_len = 輸入資料與長度
//輸出：outdata = 加解密結果，長度與輸入資料長度相同
{
    unsigned char x = m_X;
    unsigned char y = m_Y;
    for (int i=0; i < in_len; i++)
    {
        unsigned char c = ((char*)indata)[i];
        unsigned char x1 = x = (char)(x + 1);
        unsigned char y1 = y = (char)(m_State[x] + y);
        unsigned char tx = m_State[x1];
        unsigned char ty = m_State[y1];
        m_State[x1] = ty;
        m_State[y1] = tx;
        ((char*)outdata)[i] = c ^ m_State[(tx + ty) & 0xff];
    }
    m_X = x;
    m_Y = y;
}

*** 補充：block加密法

由於前文採用的RC4是stream加密法，故省略了所有block加密法的相關資訊。底下，我們以3DES為例，來進行補充說明。

    CipherSuite TLS_RSA_WITH_3DES_EDE_CBC_SHA          = { 0x00,0x0A };

以下為3DES_EDE_CBC加密法的特性：

    CipherSpec.cipher_type：stream加密法或block加密法，3DES是block加密法。
    CipherSpec.key_material：加密法的鍵值byte長度，3DES為24。
    CipherSpec.IV_size：加密法的Initial Vector byte長度，3DES為8。
    CipherSpec.hash_size：digest的hash資料byte長度，SHA-1為20。
    CipherSpec.is_exportable：是否Exportable，會影響加密的鍵值產生法，3DES_EDE_CBC為false。

block加密法和stream加密法的差異，主要是在送出Change Cipher Spec啟動加密訊息後的記錄層資料格式（參閱邏輯層第七部份）。

     block-ciphered struct {
         opaque content[SSLCompressed.length];
         opaque MAC[CipherSpec.hash_size];
         uint8 padding[GenericBlockCipher.padding_length];
         uint8 padding_length;
     } GenericBlockCipher;

    struct {
        ContentType type;
        ProtocolVersion version;
        uint16 length;
        select (CipherSpec.cipher_type) {
            case stream: GenericStreamCipher;
            case block: GenericBlockCipher;
        } fragment;
    } SSLCiphertext;

和stream加密法相較之下，block加密法會在記錄層資料的最後面，加上一些Padding資料，以使得整個資料的長度符合加密法所需
的block大小。以3DES為例，它的block大小為8，因此假設內容與MAC的大小總合為0x81，那麼後面便必須補上7個byte，以便成為8的倍數。
由於padding_lenth固定佔用1 byte，因此padding_length的值便是6。亦即當內容與MAC的大小剛好是block大小的倍數，後面還是得補上
一個block大小的資料。至於padding裡面的內容值，和padding_length相同。以前例而言，便是補上7個0x06的值。
注意MAC的計算並不含這些Padding資料。

在接收到對方的加密資料時，首先要檢查資料長度是否為block大小的倍數，如果不是便是有問題。接著解密後，將附加在最後的Padding資料
去掉即可（讀取最後一個byte便可得知padding_lenth）。

其實將SSL主架構寫好後，其他都是在擴充採用的加密法而已，並不會再動到其他地方。

*** 3DES

3DES，又稱Triple DES，是一種利用3組DES密碼來進行多重加密的方法，以補強DES加密法的弱點。一般3DES加密法的做法，
多採用EDE（Encrypt-Decrypt-Encrypt）的方式進行，其定義為：

    cipher_text = E(k3,D(k2,E(k1,plain_text)))
    plain_text = D(k3,E(k2,D(k1,cipher_text)))

其中k1,k2,k3即為使用的3組DES密碼，函數E與D分別表示加密與解密。亦即在做3DES-EDE加密時，先用第一組密碼對原始資料加密，
將結果交給第二組密碼做解密，再將結果交給第三組密碼做加密，得到最終的加密結果。解密時，反其道而行即可。

前述的定義是針對個別的block資料做加解密，前後的block資料之間並不互相影響，一般稱為ECB模式（Electronic CodeBook）。
若需要前後文相關來強化加密效果，則多會採用CBC模式（Cipher-Block Chaining），
此時會需要一個初始向量（Initialization Vector，簡稱IV）。IV在加密過程裡，主要記錄前一個block的加密結果
（只有第一個block才是真的初始向量），當下一個block要做加密時，先與IV（前一個block加密結果）做互斥後，再進行加密。

以下為CBC模式的處理流程（對每一個block依次進行）：

(1) 加密

    實際資料 = 原始資料 ^ IV
    加密結果 = 加密(實際資料)
    IV = 加密結果
    輸出加密結果

(2) 解密

    下一個IV = 已加密資料
    解密結果 = 解密(已加密資料)
    實際結果 = 解密結果 ^ IV
    IV = 下一個IV
    輸出實際結果

DES的密碼長度，主要是56 bit（7 byte），然後在內部處理時，以每7 bit共拆成8份，每份的最低位元（LSB）補上一個奇位元（Odd Parity），
最後形成8 byte的密碼。按3DES是採用3組DES密碼，故密碼長度應為168 bit（21 byte），然而實際運用時，多採用24 byte密碼，
亦即每個DES密碼都是64 bit（8 byte）。這種情況，其實主要是省下了DES 7 bit拆分的工作，故在進行DES密碼處理時，應只取各
byte的前7 bit，然後將最低位元改成正確的奇位元，最後再進行key schedule的工作。這部份可參考DES加密演算法。
